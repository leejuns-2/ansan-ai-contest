# Codex 작업 지시서

## 최우선 목표

안산시 공공데이터·AI 활용 아이디어 공모전 제출용 산출물을 완성한다. 단순 앱/챗봇이 아니라 공공데이터 기반 AI 예방 행정 시스템으로 보이게 한다.

## 작업 순서

### 1. 전체 문서 읽기

다음 파일을 모두 읽고 모순되는 표현을 정리한다.

- `contest_notice.md`
- `context.md`
- `idea_brief.md`
- `service_blueprint.md`
- `data_ai_plan.md`
- `legal_boundaries.md`
- `mvp_spec.md`
- `evaluation_strategy.md`
- `risk_review.md`
- `submission_draft_v0.md`

### 2. 제출서 v1 작성

`submission_draft_v0.md`를 바탕으로 다음 구조의 제출서 v1을 작성한다.

1. 아이디어명
2. 제안 배경
3. 문제 정의
4. 서비스 개요
5. 활용 공공데이터
6. AI 분석·활용 방법
7. MVP 및 서비스 흐름
8. 개인정보 보호 및 법적 권한 경계
9. 기대효과
10. 창의성·차별성
11. 실현가능성 및 추진계획
12. 확산 가능성

출력 파일명: `submission_draft_v1.md`

### 3. 발표자료 목차 작성

15분 발표 기준으로 슬라이드 12~15장 목차를 작성한다.

필수 포함:

- 문제 제기
- 안산시 맥락
- 서비스 개념도
- 데이터 결합 구조
- AI 모델 구조
- MVP 화면 예시
- 개인정보/권한 리스크 대응
- 기대효과
- 확산 가능성
- 예상 Q&A

출력 파일명: `presentation_outline.md`

### 4. MVP 화면 상세 기획

심사위원이 한눈에 이해할 수 있는 데모 화면을 텍스트 와이어프레임으로 작성한다.

필수 화면:

1. 오늘의 위험지도
2. 위험도 상세 패널
3. 예방 개입 추천 카드
4. 관계기관 공유 리포트
5. 개입 효과 리포트

출력 파일명: `mvp_wireframes.md`

### 5. 데이터 표 보강

`data_ai_plan.md`의 데이터 후보를 다음 컬럼으로 재정리한다.

- 데이터명
- 제공기관
- 공개 여부
- 예상 필드
- 활용 목적
- 갱신 주기
- 확보 리스크
- 대체 데이터

출력 파일명: `data_inventory.md`

주의:

- 실제 제공 여부가 불확실한 데이터는 “검증 필요”로 표시한다.
- 임의로 사실처럼 단정하지 않는다.

### 6. 리스크 대응표 작성

`risk_review.md`와 `legal_boundaries.md`를 바탕으로 심사위원 예상 질문과 답변을 만든다.

출력 파일명: `judge_qna.md`

필수 질문:

- 해경 업무와 중복 아닌가?
- 개인정보 침해 아닌가?
- AI가 꼭 필요한가?
- 실제 데이터가 있나?
- 사고 감소 효과를 어떻게 입증하나?
- 안산시가 실제로 운영할 수 있나?

### 7. 표현 정제

전체 문서에서 다음 표현을 찾아 수정한다.

수정 대상:

- 고위험자 추적
- 개인 식별
- 출입 통제 자동화
- 구조 지휘
- 위험 인물 탐지
- CCTV 감시

대체 표현:

- 고위험상황
- 고위험 군집·행동 패턴
- 익명 상황인식
- 예방 행정 지원
- 관계기관 공유
- 비식별 집계 신호

## 품질 기준

최종 문서는 다음 질문에 모두 답해야 한다.

1. 왜 안산시인가?
2. 왜 지금 필요한가?
3. 왜 공공데이터가 필요한가?
4. 왜 AI가 필요한가?
5. 왜 단순 앱/챗봇보다 창의적인가?
6. 안산시가 실제로 할 수 있는가?
7. 개인정보와 권한 문제를 어떻게 피하는가?
8. MVP가 눈에 보이는가?
9. 효과를 어떻게 측정하는가?
10. 다른 지역으로 확산 가능한가?

## 금지 사항

- 근거 없는 수치 생성 금지
- 법적 권한 단정 금지
- 실제 데이터 제공 여부를 확인하지 않고 확정 표현 금지
- 개인 식별형 AI처럼 보이는 문장 금지
- 시민 앱만 강조 금지
- 기술 용어 과시 금지
