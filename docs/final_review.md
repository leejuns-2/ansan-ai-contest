## 현재 완성도 판단

아이디어 방향은 유지해도 된다. 강점은 `사후 대응`이 아니라 `사전 예방 행정`으로 정의한 점이다. 단순 앱/챗봇보다 공공데이터와 AI 활용성이 선명하다.

다만 최종 제출 전에는 실제 안산 지역 근거와 데이터 필드 검증을 반드시 보강해야 한다. 현재 문서에는 근거 없는 수치를 단정하지 않고 `검증 필요`로 처리했기 때문에 제출 전 조사 단계로 넘길 수 있다.

## 심사항목별 점검

| 심사항목 | 현재 대응 | 보완 필요 |
|---|---|---|
| 공공데이터 활용 | 조위, 기상, 연안사고, 유동인구, 공간정보 결합 구조 제시 | 각 데이터의 최신 필드, 안산 필터 가능 여부 확인 |
| AI 활용 | 위험도 예측, 이상탐지, 익명 상황인식, 개입 추천, 효과 피드백 제시 | MVP 산식 또는 샘플 결과표 추가 필요 |
| 기획력 | 예방 행정 시스템으로 문제 정의 명확 | 안산시 정책·관광·안전계획과 연결하면 강화 |
| 효과성 | 선행 KPI 제시 | 실제 사고·민원·체류 데이터 확보 시 정량화 필요 |
| 창의성 | 행정 개입 추천과 피드백 루프가 차별점 | MVP 화면 이미지 또는 데모가 있으면 발표력 강화 |
| 실현가능성 | 공개데이터 MVP → 시범구간 → 확대 단계화 | 운영부서, 전광판/안내채널 연동 가능성 확인 |

## 표현 리스크 점검

### 사용해도 되는 표현

- 고위험상황
- 고위험 군집·행동 패턴
- 익명 상황인식
- 비식별 집계 신호
- 예방 행정 지원
- 관계기관 공유
- 위험 노출 감소
- 설명 가능한 위험도 모델

### 피해야 하는 표현

- 고위험자 추적
- 개인 식별
- 위험 인물 탐지
- CCTV 감시
- 개인별 위험점수
- 출입통제 자동화
- 구조 지휘
- AI가 사고를 완전히 막는다

### 현재 산출물 상태

신규 산출물에서는 위험 표현을 대부분 제외하거나 `제외 기능`, `금지 표현`, `주의사항`으로만 사용했다. 최종 제출서로 옮길 때는 금지 표현 목록 자체가 아니라 권장 표현 중심으로 정리하는 편이 좋다.

## 법적·권한 경계 점검

핵심 원칙:

```text
안산시는 위험 예측, 시민 안내, 시설 개선, 현장 안내 강화, 관계기관 공유를 수행한다.
해양경찰 등 관계기관의 구조·통제·출입통제 권한은 대체하지 않는다.
```

최종 제출서에 반드시 포함할 문장:

```text
본 시스템은 해양경찰 등 관계기관의 법정 권한을 대체하지 않는다. 안산시는 공공데이터와 AI를 활용해 위험 시간·장소·상황을 사전에 예측하고, 시민 안내·시설 개선·현장 안내 강화·관계기관 공유 등 지방자치단체가 수행 가능한 예방 행정 조치를 지원한다.
```

추가 확인:

- 연안사고 예방에 관한 법률 시행규칙 제5조 관련 조문 최신 확인
- 안산시가 운영 가능한 안내채널, 전광판, 관광안내 시스템 범위 확인
- 관계기관 공유 리포트의 실제 전달 프로세스 확인

## 개인정보 보호 점검

현재 설계는 개인 식별을 배제하는 방향으로 잡혀 있다.

포함된 보호 원칙:

- 얼굴인식 미사용
- 차량번호 인식 미사용
- 개인별 이동경로 저장 금지
- 개인별 위험점수 부여 금지
- 원본 영상 미저장 또는 최소화
- 구역 단위 인원 수·방향·체류시간만 활용

추가 확인:

- 현장 CCTV/IoT를 실제로 쓰는 경우 개인정보 영향평가 필요 여부
- 엣지 단말 처리 후 저장되는 데이터 항목
- 원본 영상 보관 여부와 보관 기간
- 내부 접근 권한과 로그 관리

## MVP 보강 제안

현재 텍스트 와이어프레임은 제출서와 발표 구조를 잡기에는 충분하다. 발표 경쟁력을 높이려면 다음 중 하나를 추가하는 것이 좋다.

1. 샘플 CSV 1개
   - 장소 3곳, 시간대 12개, 위험도와 원인 Top 3 포함

2. 위험도 산식 예시
   - `risk_score = 조위위험 + 기상위험 + 혼잡위험 + 사고이력 + 공간위험`
   - 가중치는 임의값임을 표시

3. 대시보드 목업 이미지
   - 지도, 위험도 상세, 추천 카드, 효과 리포트를 한 화면 흐름으로 표현

## 제출 전 우선순위

1. 안산 연안 사고·위험 사례 공식 근거 확보
2. 데이터 인벤토리의 공개 여부와 필드 최신 확인
3. 제출서 v1을 신청서식 분량과 문항에 맞게 압축
4. 발표자료 목차를 실제 PPT로 전환
5. MVP 화면 2~3장 목업 제작
6. 예상 Q&A를 발표자 답변 문장으로 암기 가능한 형태로 축약

## 최종 판단

현재 상태는 `공모전 제출안 작성에 들어갈 수 있는 설계 완료` 단계다. 다음으로는 문서 작성보다 근거 보강과 시각화가 중요하다.

가장 큰 남은 리스크는 세 가지다.

1. 안산 특정 장소의 실제 사고·위험 근거 부족
2. 유동인구와 현장 신호 데이터의 해상도 불확실성
3. 발표에서 CCTV나 통제 시스템처럼 오해받을 가능성

이 세 가지를 관리하면 제안의 핵심 구조는 충분히 방어 가능하다.
